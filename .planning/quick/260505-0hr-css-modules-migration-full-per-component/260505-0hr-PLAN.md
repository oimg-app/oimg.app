<conventions>
- Module file naming: lowercase camelCase matching component (`titleBar.module.css`).
- Import: `import s from './titleBar.module.css'` then `<div className={s.titlebar}>`.
- Multi-class: prefer `clsx` (already in package.json — `"clsx": "^2.1.1"`). Use `clsx(s.tbtn, isOpen && s.open)`.
- Shared primitives: declare local class with `composes: pane from '@/styles/primitives.module.css'` rather than importing primitive class directly into JSX. Components reference their LOCAL class only.
- Class-name policy in CSS source: keep kebab-case names AND add explicit camelCase aliases when the name has a hyphen — Vite's CSS Modules auto-camelCases by default but be explicit to avoid surprises. When a class is `pane-hd` in CSS, expose `paneHd` (use `.paneHd` directly in CSS or rely on auto-conversion — pick auto-conversion + camelCase keys in JSX consistently across all components).
- Pseudo-element selectors (`input[type=range]::-webkit-slider-thumb`, `:focus-visible`, etc.) wrap in `:global()` inside the module so the selector matches raw DOM elements.
- Token CSS vars (`--bg`, `--fg`, `--accent`, etc.) stay global — do NOT scope through modules.
</conventions>

Notes spotted while reading:
- **SvgoPanel** uses `.section`, `.section-title`, `.plugin`, `.plugin-check` — these are SHARED primitives → live in `primitives.module.css`. SvgoPanel composes them.
- **CodecPanel + TweaksPanel** also use `.section` / `.section-title` and `.sliderBlock` / range inputs — same primitives.
- **SnippetPanel** uses `.code` + `.code-row` + `.copyBtn` (shared) AND its own `.tok-*` syntax-highlight tokens (PRIVATE → live only in `snippetPanel.module.css`).
- **ContextMenu** `.ctxbtn` is positioned by its parent `.file-row` (in QueuePane). Parent-child class boundary: ContextMenu owns `.ctxbtn` styles but the parent must apply `.file-row` (from `queuePane.module.css`) to the `<li>` so `:hover` reveal of `.ctxbtn` works. The reveal selector lives in `queuePane.module.css` as `.fileRow:hover .ctxbtn` using `:global(.ctxbtn)` OR by exporting the ctxbtn class via a composes chain. Recommended: in `contextMenu.module.css` define `.ctxbtn` and ALSO declare `:export { ctxbtn: ctxbtn; }` — then in `queuePane.module.css` import it: `composes: ctxbtn from '../../file-row/ContextMenu/contextMenu.module.css'` is NOT valid for descendant selectors. Instead: use `:global` for the descendant selector inside `queuePane.module.css`:
  ```css
  .fileRow :global(.ctxbtn-trigger) { opacity: 0; }
  .fileRow:hover :global(.ctxbtn-trigger) { opacity: 1; }
  ```
  ContextMenu adds a stable global marker class `ctxbtn-trigger` alongside its module class on the trigger button. This is the cleanest way to handle parent-driven hover reveal across module boundaries.
- **AppShell** owns the grid (`.app`); TitleBar/Toolbar/work-area/StatusBar are slot children — AppShell does NOT style them, only places them. Shell row classes (`.titlebar`, `.toolbar`, `.statusbar`) live in those components' modules.
- **CommandPalette** uses `.kbd` (shared) + its own `.cmdk-*` set (private).
- **Popover** primitive `.popover`, `.popover-item`, `.popover-divider`, `.popover-label` are SHARED → live in `primitives.module.css` (consumed by Popover module + any component that renders raw popover markup).
