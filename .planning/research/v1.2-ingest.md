# v1.2 Research — From URL / Paste Ingest

**Goal:** Add two new ingest channels alongside the existing device picker + watch folder: **paste from clipboard** (Cmd/Ctrl+V) and **from URL** (string or dragged-link). The flows must extend `useIngest` (single ingestion seam — D-01 from Phase 10) rather than create a parallel pipeline.

**Confidence:** HIGH on browser API behavior, HIGH on CORS limitations (these are physical browser constraints, not opinion).

---

## TL;DR — Honest Recommendation

- **Paste** works reliably for image bytes (image off the clipboard) and for URL strings (re-routed to the URL flow). Use a **document-level `paste` event handler** instead of `navigator.clipboard.read()`; it works in non-secure contexts and avoids a permission prompt.
- **URL fetch** is fundamentally CORS-bounded. There is **no real bypass** for byte access from a browser without a server. Best-effort flow:
  1. Try `fetch(url)` direct — works only if the origin sends `Access-Control-Allow-Origin: *` (or matches).
  2. On failure, toast a clear error: "Server blocked cross-origin access. Download the image and drop it here."
- **Do NOT** ship a `<img crossorigin> + canvas.toBlob()` fallback. It re-encodes the image (losing format and quality) and silently fails on tainted canvases — worse UX than the toast.

## 1. Paste Flow (`pickFromClipboard`)

### API choice

| API | Secure-context only? | Works for images? | Verdict |
|---|---|---|---|
| `navigator.clipboard.read()` | YES (HTTPS or localhost) | YES (ClipboardItem) | Requires user permission grant; overkill |
| `document.addEventListener('paste', e => e.clipboardData.items)` | NO | YES | **PICK** — works everywhere, no permission |

### Implementation sketch

```ts
// src/hooks/useClipboardIngest.ts
import { useEffect } from 'react'
import { useIngest } from '@/hooks/useIngest'
import { pickFromUrl } from '@/lib/url-ingest'

export function useClipboardIngest() {
  const { ingest } = useIngest()
  useEffect(() => {
    async function onPaste(e: ClipboardEvent) {
      // Ignore paste inside text inputs / textareas (let the browser handle it)
      const target = e.target as HTMLElement | null
      if (target && /^(input|textarea)$/i.test(target.tagName)) return
      const items = e.clipboardData?.items
      if (!items?.length) return

      const files: File[] = []
      let urlText: string | null = null

      for (const item of items) {
        if (item.kind === 'file' && item.type.startsWith('image/')) {
          const f = item.getAsFile()
          if (f) files.push(f)
        } else if (item.kind === 'string' && item.type === 'text/plain') {
          urlText = await new Promise<string>(r => item.getAsString(r))
        }
      }

      if (files.length > 0) {
        e.preventDefault()
        await ingest(files)
        return
      }
      if (urlText && /^https?:\/\/.+\.(png|jpe?g|webp|svg|avif)(\?|$)/i.test(urlText)) {
        e.preventDefault()
        const fetched = await pickFromUrl(urlText.trim()).catch(() => null)
        if (fetched) await ingest([fetched])
      }
    }
    document.addEventListener('paste', onPaste)
    return () => document.removeEventListener('paste', onPaste)
  }, [ingest])
}
```

Mount once at AppShell level. WCAG-AA note: a screen-reader-only `<div>` with `aria-live="polite"` announcing "1 image pasted" (via toast) keeps parity with file-drop UX.

### Gotchas

- **Safari 17+** delivers PNG when an image was copied from Preview; **Chrome** may deliver a screenshot as `image/png` even when the source was JPEG (since clipboard normalizes to PNG). The codec gate accepts both, no special handling needed.
- **Firefox** historically returned the file with name `image.png`. Our store generates UUIDs and uses `file.name`, so this is fine.
- **iOS Safari** lacks the paste event in most contexts; mobile is best-effort, not a v1.2 requirement.

## 2. URL Flow (`pickFromUrl`)

### The CORS reality

There is **no JS-only way** to read bytes of an arbitrary cross-origin image without server cooperation. Concretely:

| Approach | Bytes accessible? | Notes |
|---|---|---|
| `fetch(url)` (CORS) | YES if server allows | Most CDN-served images allow it; user content rarely does |
| `fetch(url, { mode: 'no-cors' })` | NO | Returns opaque response — can't read bytes |
| `<img crossorigin="anonymous">` + canvas + `toBlob()` | YES if server sends ACAO + we re-encode | Re-encoding **destroys** original format; canvas taints fail otherwise |
| `<img>` without `crossorigin` | NO (canvas tainted) | DOM render works; readback throws SecurityError |
| Proxying through our own origin | YES | **Violates the zero-server constraint** — not an option |

**Decision: do not ship the canvas fallback.** Re-encoding a PNG into our pipeline as a PNG is honest only by coincidence; if the source was JPEG, we'd silently change the format and lose EXIF. The user expects "we optimize *this* file" — re-encoding behind their back breaks trust.

### Implementation sketch

```ts
// src/lib/url-ingest.ts
const MAX_URL_BYTES = 100 * 1024 * 1024  // 100 MB hard cap

export async function pickFromUrl(url: string): Promise<File> {
  // Validate scheme — only http/https
  const parsed = new URL(url)  // throws TypeError on garbage
  if (!/^https?:$/.test(parsed.protocol)) {
    throw new Error('Only http(s) URLs are supported')
  }

  // Try direct fetch — only path that gets us real bytes
  const res = await fetch(url, { mode: 'cors', credentials: 'omit' })
  if (!res.ok) throw new Error(`Server returned ${res.status}`)
  const ct = res.headers.get('content-type') ?? ''
  if (!ct.startsWith('image/')) throw new Error(`Not an image (content-type: ${ct})`)

  const cl = Number(res.headers.get('content-length') ?? 0)
  if (cl > MAX_URL_BYTES) throw new Error('Image is too large (max 100 MB)')

  const blob = await res.blob()
  if (blob.size > MAX_URL_BYTES) throw new Error('Image is too large (max 100 MB)')

  // Derive filename from URL path or content-disposition
  const cd = res.headers.get('content-disposition') ?? ''
  const cdMatch = cd.match(/filename="?([^"]+)"?/)
  const name = cdMatch?.[1] ?? parsed.pathname.split('/').pop() ?? 'pasted-image'

  return new File([blob], name, { type: blob.type, lastModified: Date.now() })
}
```

### Surfacing the failure clearly

`Toolbar.tsx` "From URL or paste" stub currently calls `addFromUrl()` (an empty stub at `src/stores/files.ts:88`). Replace with a small modal/popover that:
1. Accepts a URL string
2. Calls `pickFromUrl(url)` → `ingest([file])`
3. On `TypeError` (failed fetch — almost certainly CORS): toast(`Server blocked cross-origin access. Download the image and drop it here.`)
4. On any other error: toast the error message

```tsx
async function onSubmit(url: string) {
  try {
    const file = await pickFromUrl(url)
    await ingest([file])
  } catch (err) {
    const msg = err instanceof TypeError
      ? 'Server blocked cross-origin access. Download the image and drop it here.'
      : (err instanceof Error ? err.message : String(err))
    toast.error(msg)
  }
}
```

## 3. Drag-from-tab (HTML fragment) — Stretch

When users drag an image from another browser tab, `dataTransfer.getData('text/html')` exposes the image's source URL inside an `<img>` element. This is just URL ingest with extraction; reuse `pickFromUrl`.

```ts
function extractImgSrc(html: string): string | null {
  const match = html.match(/<img[^>]+src=["']([^"']+)["']/i)
  return match?.[1] ?? null
}

// In FilesPane drop handler:
const html = e.dataTransfer.getData('text/html')
const url = html ? extractImgSrc(html) : null
if (url) { /* same pickFromUrl flow as above */ }
```

Same CORS constraint applies — most cross-tab drags will fail. Surface the same error message.

`useClipboardIngest` is one-way: it pipes into the existing `ingest()`. `pickFromUrl` is one-way: returns a `File` that the caller passes to `ingest()`. **No new ingestion seams** — preserves the D-01 Phase 10 invariant.

## 6. Failure Modes Documented

| Scenario | What user sees | Tech reason |
|---|---|---|
| Paste an image directly | File appears in queue | `clipboardData.items` exposes the blob |
| Paste a URL to allowing CDN | File appears in queue | `fetch` succeeds |
| Paste a URL to blocking server | Toast: "Server blocked cross-origin access. Download the image and drop it here." | `fetch` rejects with `TypeError` |
| Paste a URL with auth-required image | Toast with HTTP status (401/403) | `fetch` succeeded but `res.ok` is false |
| Paste a URL pointing to non-image | Toast: "Not an image" | content-type check |
| Drag image from another tab (CORS-friendly) | File appears | `pickFromUrl` succeeds |
| Drag image from another tab (most cases) | Toast: same CORS error | Same CORS failure as URL flow |
