**What:** All design tokens are defined in `src/index.css` using `@theme inline {}` blocks and CSS custom properties. There is NO `tailwind.config.ts` in Tailwind v4.

- **Writing `tailwind.config.ts`:** Tailwind v4 ignores it. All config goes in CSS `@theme` blocks.
- **Importing stub-data directly in components:** Violates STORE-08. Components use `useStore()` on atoms; atoms initialize from stub-data. Only stores import stub-data.
- **CSS Modules:** Deleted in refactor. All styling via Tailwind utility classes on JSX elements.
- **Inline styles:** Forbidden per CLAUDE.md conventions.
- **`useState` for data:** Only ephemeral hover/focus state allowed in components per STORE-08.
- **Skipping `npm install`:** node_modules does not exist. First step in every plan must be `npm install`.

## Common Pitfalls

### Pitfall 1: Tailwind v4 has no JS config file
**What goes wrong:** Developer creates `tailwind.config.ts` expecting v3 behavior; tokens don't apply.
**Why it happens:** Muscle memory from Tailwind v3; documentation search returns v3 results.
**How to avoid:** All token configuration goes in `src/index.css` via `@theme inline {}` blocks. The `@tailwindcss/vite` plugin reads CSS imports, not a config file.
**Warning signs:** Custom colors like `bg-accent` not working in dev server.

### Pitfall 2: node_modules missing — app won't start
**What goes wrong:** `npm run dev` fails with module not found errors.
**Why it happens:** The working tree has only source files; the refactor deleted everything and node_modules was never reinstalled.
**How to avoid:** Every plan wave must include `npm install` as the first step.
**Warning signs:** `vite: command not found` or `Cannot find module 'react'`.

### Pitfall 3: Dark theme — wrong class vs. data-attribute selector
**What goes wrong:** Dark theme doesn't apply; components render in light theme even when dark is intended.
**Why it happens:** The prior `src/index.css` uses `.dark {}` CSS selector (class-based), but OIMG.html uses `[data-theme="dark"]`. These are different approaches.
**How to avoid:** The prior codebase used Tailwind's dark mode via `.dark` class + `@custom-variant dark (&:is(.dark *))`. Phase 1 `AppShell` should add `className="dark"` to the root div or `<html>` for the default dark theme. Theme switching (Phase 3/7) will use the same class toggle.
**Warning signs:** Background is white/light when it should be `oklch(0.165 0.008 250)`.

### Pitfall 4: Shadcn resizable.tsx depends on react-resizable-panels imports
**What goes wrong:** Shadcn's generated `resizable.tsx` imports from `react-resizable-panels`. If the package isn't installed, the component fails silently or throws.
**Why it happens:** `react-resizable-panels` is a runtime dependency (in `dependencies`, not `devDependencies`).
**How to avoid:** `npm install` resolves it. Verify the import in `resizable.tsx` matches `react-resizable-panels@4.11.0` API.
**Warning signs:** TypeScript errors on `PanelGroup`, `Panel`, or `PanelResizeHandle` imports.

### Pitfall 5: `@import "@fontsource-variable/inter"` location
**What goes wrong:** Inter font not loading; fallback system-ui shows instead.
**Why it happens:** The import is in `src/main.tsx`, not `src/index.css`. If `App.tsx` is rebuilt without checking `main.tsx`, the font import might be omitted.
**How to avoid:** Keep the font imports in `src/main.tsx` (they are already there). Don't move them.
**Warning signs:** Text renders in system-ui/sans-serif instead of Inter.

### Pitfall 6: stub-data types conflict with Phase 2+ store types
**What goes wrong:** `FileEntry` defined in `stub-data.ts` conflicts with the richer `FileEntryWithBlob` defined in stores in Phase 2+.
**Why it happens:** Phase 1 `FileEntry` is a UI-display-only shape; Phase 2 adds blob storage fields.
**How to avoid:** Define `FileEntry` as the minimal display shape in `stub-data.ts`. Stores in Phase 2 will extend it. Export the type from `stub-data.ts`; phases 2+ re-export or extend it.

### format.ts (verbatim from git HEAD — verified correct)

```typescript
export function fmtBytes(b: number | null | undefined): string {
  if (b == null) return '—';
  if (b < 1024) return b + ' B';
  if (b < 1024 * 1024) return (b / 1024).toFixed(1) + ' KB';
  return (b / 1024 / 1024).toFixed(2) + ' MB';
}

export function fmtPct(orig: number, opt: number): string {
  if (!orig || !opt) return '—';
  const saved = ((orig - opt) / orig) * 100;
  if (saved === 0) return '';
  return (saved > 0 ? '−' : '+') + Math.abs(saved).toFixed(1) + '%';
}
```
