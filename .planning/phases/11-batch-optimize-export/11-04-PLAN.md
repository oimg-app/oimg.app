<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `entry.name` (user file system) → `showSaveFilePicker.suggestedName` | Browser sanitizes invalid filesystem chars at the OS layer when writing; `renameExtension` (Plan 03) already normalizes; Plan 05 adds `sanitizeBaseName` for the ZIP path. Single-file picker presents the name to the user for confirmation — additional defense-in-depth optional |
| `entry.encodedBuffer` (worker output) → `Blob` constructor | Trusted: bytes come from the codec worker, no untrusted post-processing |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-PIC | Information Disclosure | showSaveFilePicker on non-secure context (http://) throws | mitigate | Task 1 gates with `window.isSecureContext === true`; insecure context falls through to file-saver fallback silently |
| T-11-ABT | Information Disclosure | AbortError leaking to toast/console reveals user intent + creates noise | mitigate | Task 1 silently swallows AbortError; no fall-through to fallback (would re-prompt); Test 3 in Task 3 asserts no toast appears |
| T-11-OPQ | Information Disclosure | Opaque picker errors (e.g. sandboxed iframe) surfacing to user | mitigate | Task 1 catches ALL non-AbortError exceptions and falls through to file-saver (user still gets their file) |
| T-11-SUG | Tampering | Malicious source filename (`../../etc/passwd`) in suggestedName | accept | Browser-side: `showSaveFilePicker` itself sanitizes invalid path chars before presenting the dialog; user reviews the suggested name before save. Defense-in-depth via Plan 05's `sanitizeBaseName` for the ZIP path |
</threat_model>
