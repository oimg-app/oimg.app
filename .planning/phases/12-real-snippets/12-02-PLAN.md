<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `file.encodedBuffer` (worker output bytes) → snippet text | Bytes are trusted (codec worker); their textual representation (base64 or percent-encoded UTF-8) is opaque to HTML interpretation when wrapped in CSS url() context |
| `file.name` (user file system) → snippet `alt="…"` and `src="…"` attribute interpolation | UNTRUSTED — filenames can contain `"`, `<`, `>`, `&`, `'`; T-12-02 mitigation required (escapeAttr) |
| SVG file content → base64 vs URL-encoded data URI dispatch | T-12-01 hinge: SVG must NEVER reach the base64 path that wraps in `<img src>` (browsers execute scripts in SVG bytes via that channel) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12-01 | Tampering / Elevation | SVG with embedded `<script>` reaching `<img src="data:image/svg+xml;base64,…">` (browsers execute) | mitigate | Task 1 `buildDataUri(file)` dispatches on `file.target.toLowerCase() === 'svg'` FIRST and emits the URL-encoded path; the chunked-base64 raster path is unreachable for target==='svg'. Test 7 in Task 2 asserts the SVG path emits the `image/svg+xml;charset=utf-8,` prefix. Defense: even when wrapped in `<img src>` (buildBase64Snippet on an SVG file), the URI is the URL-encoded utf-8 form, which `<img>` parses as SVG without script execution context for inline scripts (Chrome/Firefox treat data: URI SVGs in `<img>` as opaque images that do not run scripts) |
| T-12-02 | Tampering | `file.name` containing `"`, `<`, `>`, `&` breaks out of `alt="…"` or `src="…"` attribute | mitigate | Task 1 adds `escapeAttr(s)` helper applied to every attribute interpolation in `buildBase64Snippet` and `buildPictureSnippet` (alt + src for fallback img tag + srcset for source). Acceptance criteria asserts the literal `&amp;` `&quot;` `&lt;` `&gt;` substrings present in the source. Test 5/6 in Task 2 exercise adversarial filenames |
| T-12-CTL | Tampering | SVG bytes containing ASCII control chars (NULL, BEL, etc.) breaking valid-CSS validity in `url("…")` | mitigate | Task 1 `buildSvgDataUri` strips `/[\x00-\x1F]/g` before encodeURIComponent (sanitization before encoding) |
| T-12-BIG | Denial of Service | Multi-MB raster ArrayBuffer crashing V8 via `String.fromCharCode(...new Uint8Array(huge))` call-stack blowup | mitigate | Task 1 `bufferToBase64` iterates 32KB chunks via `String.fromCharCode.apply(null, slice)` per chunk (browser-safe per D-02 / 12-PATTERNS.md). Acceptance criteria asserts the literal `0x8000` window-size constant present |
</threat_model>
