<decisions>
## Plan-Level Decisions

### Hand-rolled primitives stay; shadcn migration deferred
The current codebase ships hand-rolled `Popover`, `Tooltip`, `Slider`, `Toggle`, `Seg`, and `Section` components under `src/components/ui/`. CONTEXT.md D-06 calls for shadcn primitives. The divergence briefing offered two paths: (a) replace primitives with shadcn now, (b) accept the deviation and migrate later.

We pick (b). Reasons:
- The hand-rolled primitives already render correctly under all 4 inspector panels and are visually faithful to example-ui/OIMG.html (the visual contract D-07).
- Replacing them now is invasive (every panel + App.tsx menu/popover usage), risks visual regression, and burns time without changing user-visible behavior or unblocking Phase 2.
- Phase 1 success criteria (per ROADMAP.md) require the shell, COOP/COEP, theme toggle, and keyboard/ARIA — none of these depend on whether Popover is hand-rolled or Radix-backed.
- WCAG AA risks of hand-rolled primitives are real (focus management, Escape handling, role attributes). This plan adds Playwright tests that verify the EXISTING hand-rolled implementations meet the keyboard/ARIA requirements. If a test fails, we fix the specific primitive — but we do NOT pre-emptively swap them all out.

A new entry is added to `.planning/phases/01-shell-foundation/deferred-items.md` so Phase 8 (Polish + Deploy) picks up the shadcn migration as part of the WCAG AA audit.

### App.tsx work-area JSX stays in App.tsx
The original 05-PLAN expected three new components: FilePanel, DetailPanel, TweaksPanel. Reality: the work area in App.tsx renders three panes (queue, compare, inspector with tab switcher) and the inspector tab bodies (CodecPanel/SvgoPanel/OutputPanel/ReportPanel) are already extracted as the actual "panels" of the app. Re-extracting work-area JSX into FilePanel/DetailPanel/TweaksPanel adds files without changing behavior. We instead invest the budget in: making the work-area DOM properly ARIA-labeled (listbox/tablist/slider) and writing tests that prove it. Re-extraction is a Phase 2 concern when state moves to Zustand stores and panel boundaries get useful prop interfaces.
</decisions>
